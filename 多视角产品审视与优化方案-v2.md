# 多视角产品审视与优化方案 v2

更新时间：2026-02-28  
范围：`项目A-求职面试复盘助手`（Web 版）

## 1. 结论（严格产品视角）

- 当前版本已具备“可运行闭环”，并通过严格自动化回归（10/10）。
- 当前最大风险不在“能不能跑”，而在“研究质量是否稳定达标”。
- 下一阶段应把重心从“功能叠加”切换到“质量阈值治理 + 数据可信度提升”。

## 2. 四种角色审视

### 2.1 使用者视角（求职者）

核心关注：能否快速得到可执行建议、建议是否可信。  
发现问题：
- 功能很多，但用户难以判断当前系统是否“可靠可用”。
- 研究门禁未通过时，用户容易不清楚问题到底出在来源、证据还是模型。

已改进：
- 新增“项目健康看板（多角色视角）”，直接展示通过率、错误率、风险提醒。

下一步建议：
- 在每次输出策略旁增加“可执行置信度”与“最短行动路径”。

### 2.2 开发者视角（系统维护）

核心关注：线上稳定性、失败可追踪、异常可定位。  
发现问题：
- 之前缺少统一健康快照接口，不利于日常巡检。
- 闭环测试关注主流程，但缺少“项目整体状态”的健康断言。

已改进：
- 新增 `GET /api/project/health`，输出统一健康快照。
- 闭环脚本纳入项目健康快照校验，严格模式总链路提升为 10 项。

下一步建议：
- 增加定时自动回归（每天一次），并把结果写入同一日志文件。

### 2.3 程序员视角（代码与算法）

核心关注：检索解析鲁棒性、聚类正确性、数据噪声控制。  
发现问题：
- DuckDuckGo 在部分环境会触发人机挑战。
- Bing 页面结构变化导致抽取规则过窄时失效。
- 中文论点聚类过于保守时，会出现“来源很多但合格聚类为 0”。

已改进：
- 放宽 Bing HTML 解析规则（`h2/a/p` 标签结构兼容）。
- 中文 claim 改为 `2~3gram` 分词，提升聚类稳定性。
- 新增“主题论点聚类兜底”，避免语义聚类失配导致假失败。

下一步建议：
- 增加“岗位相关性过滤器”（按岗位词和公司词过滤低相关来源）。

### 2.4 产品经理视角（价值与迭代）

核心关注：价值闭环、可验证指标、迭代优先级。  
发现问题：
- 指标存在但分散，不利于快速决策。
- 当前阶段更像“能力拼图”，而非“最小可卖产品”。

已改进：
- 健康看板整合四视角状态与建议，便于版本评估与优先级决策。

下一步建议：
- 明确 P0 指标：`7天 prep 通过率`、`7天 deep research 通过率`、`证据支持率`。
- 以指标驱动迭代，不再以“功能数量”驱动迭代。

  - `app/src/lib/project-health.ts`

  - `app/scripts/closed_loop_e2e.mjs`
  - 新增 `project-health.snapshot` 检查
  - 严格门槛：`sources >= 3` 且 `acceptedClusters >= 1`
- 文档更新：
  - `app/README.md`

## 4. 当前优先级建议（按收益/成本排序）

1. P0：岗位相关性过滤（提升来源均分与门禁通过率）
2. P0：证据支持率提升机制（弱证据条目自动复查补强）
3. P1：策略输出的“行动优先级排序 + 时间成本估算”
4. P1：一键导出“面试前准备简报（可投递前复习）”

## 5. 验收标准（下一迭代）

- 严格闭环持续 `10/10`。
- `7天 deep research 通过率 >= 55%`。
- `证据支持率 >= 30%`。
- “项目健康看板”中高优先级 alerts 数量持续下降。
